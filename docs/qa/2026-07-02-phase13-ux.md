# Phase 13 — UX Review

**Date:** 2 July 2026 · **Scope:** every user-facing reply, the onboarding/approval flow, command discoverability, and message formatting in `worker/src/index.js` (plus the `setMyCommands` menu)
**Method:** walked all 15 command paths and the callback (approve/deny) flow from the perspective of each role — new/pending user, approved user, owner — and checked formatting (`parse_mode`), tone consistency, and dead-ends.

---

## Executive summary

The conversational UX is **thoughtful and consistent**: every message is answered (no silent dead-ends since BUG-3 was fixed), unapproved users always get a clear "send /start" pointer with their own Telegram ID, generation gives an immediate "one moment…" acknowledgement before the async result arrives, and the privacy commands (`/privacy`, `/mydata`, `/forgetme`) work even before approval. Tone is warm and plain, no emoji clutter in command replies.

Findings are **low-to-medium polish** issues. The one that actually bites real users daily is **case-sensitive commands** (UX-1) — mobile keyboards autocapitalize `/Start`, and the bot rejects it. The rest are cosmetic (literal backticks in admin replies), a discoverability gap (`/newbriefing` missing from `/help`), and one deliberate-but-worth-confirming silence (denied users hear nothing).

**No release blocker.**

---

## Findings

### UX-1 (Medium) — Commands are case-sensitive; mobile autocapitalization breaks them
The dispatcher looks up `COMMAND_HANDLERS[m[1]]` without lowercasing (`worker/src/index.js:753`), and all handler keys are lowercase. So `/Start`, `/Briefing`, `/Help` — exactly what iOS/Android produce when autocapitalize is on and the user types the command by hand — fall through to the generic "I only understand commands" nudge. The worst case is a **brand-new user**: their client capitalizes `/Start`, they get "send /start to request access," they're confused about why the thing they just sent didn't work. Fix: `const name = m ? m[1].toLowerCase() : null` before the lookup (Telegram commands are case-insensitive by convention, and BotFather's own menu sends lowercase, so this only helps and never hurts).

### UX-2 (Low–Medium) — `/help` omits `/newbriefing` (menu/help mismatch)
The `/help` body lists `/briefing`, `/subscribe`, `/unsubscribe`, `/status`, `/privacy`, `/mydata`, `/forgetme` (`worker/src/index.js:447-453`) but **not** `/newbriefing` — even though `setMyCommands` registers `/newbriefing` in the client menu (`scripts/set-commands.mjs:21`). So the command exists, shows up in the "/" autocomplete, but is undocumented in the one place a user goes to learn what the bot does. Either add a `/newbriefing — force a fresh briefing now` line to `/help`, or (given `/briefing` already regenerates when the cache is stale) consider whether `/newbriefing` needs to be user-facing at all. At minimum, make help and the menu agree.

### UX-3 (Low) — Admin replies render literal backticks
Four owner-facing messages wrap the chat id in Markdown backticks but send as **plain text** (no `parse_mode`), so the owner literally sees `` `123456` `` with the backtick characters:
- `worker/src/index.js:636` — "User `` `id` `` is already on the allowlist."
- `:639` — "User `` `id` `` added to the allowlist."
- `:660` — "User `` `id` `` not found."
- `:663` — "User `` `id` `` removed…"
Cosmetic and owner-only, but sloppy. Fix: drop the backticks, or switch these to `<code>${id}</code>` with `{ parse_mode: 'HTML' }` (the `/status` and `/start` replies already do this correctly).

### UX-4 (Low) — Denied users are left in silence
On deny, the callback removes the pending entry, answers the owner's callback, and edits the owner's message — but sends **nothing to the applicant** (`worker/src/index.js:732-736`). The person who sent `/start` waits indefinitely with no signal. This may be intentional (don't advertise rejection), but it's a UX decision worth making explicitly: a neutral "Your access request wasn't approved at this time" closes the loop without revealing much. If the silence is deliberate, document it; otherwise add the courtesy reply.

### UX-5 (Low, nit) — "Paired" vocabulary is a leftover from the removed pairing-code model
`/status` says "Paired as @name" and approval says "Paired!" (`worker/src/index.js:460,731`), but the bot is **allowlist**-based — the source itself notes "No pairing-code machinery here" (`:329`). "Paired" implies a device/code handshake that doesn't exist; "You're approved" / "Approved as @name" would match the actual model. Purely wording consistency.

---

## Strengths (verified)

- **No dead-ends:** plain text, typos, non-text messages, and prototype-name commands all get a helpful nudge (post-BUG-3), tailored to whether the sender is approved.
- **Self-service identity:** every unapproved-state reply includes the user's own `<code>Telegram ID</code>`, which is exactly what they need to give the owner or use with `/adduser`.
- **Async feedback done right:** `/newbriefing` and stale-cache `/briefing` immediately acknowledge ("Generating…, one moment") before the runner delivers the result, and the cooldown path serves the cached briefing with an explanation rather than a bare refusal.
- **Privacy commands ungated:** `/privacy` / `/mydata` / `/forgetme` work pre-approval — a notice you can't read until approved wouldn't be a notice.
- **Approval ergonomics:** inline Approve/Deny buttons, the owner's message is edited to show the outcome, and the approved user gets an immediate "Paired! Send /briefing or /subscribe" onboarding nudge.

## Summary

| ID | Sev | Finding | Fix |
|---|---|---|---|
| UX-1 | Medium | Case-sensitive commands; mobile autocapitalize breaks `/Start` etc. | `.toLowerCase()` the captured command name |
| UX-2 | Low–Med | `/help` omits `/newbriefing` (menu lists it, help doesn't) | Add the line, or drop the command from the menu |
| UX-3 | Low | Admin id replies show literal backticks (no `parse_mode`) | Remove backticks or use `<code>` + HTML |
| UX-4 | Low | Denied applicants get no notification | Send a neutral close-the-loop reply, or document the silence |
| UX-5 | Low | "Paired" wording survives from the removed pairing model | Reword to "approved" |

**Net:** the core conversational flow is genuinely good — nothing leaves the user stuck or unanswered, onboarding is self-explanatory, and async operations acknowledge immediately. UX-1 is the only finding with everyday impact (and it's a one-line fix); the rest are cosmetic polish. No blocker to release.
